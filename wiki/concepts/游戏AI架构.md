# 游戏AI架构设计


## 概述

游戏AI架构是决定AI系统成败的基石。从"行为选择算法"和"结构架构"两个维度，系统介绍了不同规模和类型的游戏应该选择什么样的AI架构。

架构设计的本质问题是：

> **AI 如何从众多可能行为中做出选择？这些选择逻辑如何组织在代码中？**

本技能提供架构选型的决策框架和常见模式的实战指南。

### 适用场景

- 新项目启动时选择AI技术栈
- 现有AI系统遇到扩展瓶颈需要重构
- 团队需要统一AI开发规范
- 跨平台移植时评估AI性能方案

## 核心概念

### 行为选择算法分类

将游戏AI的行为选择算法分为以下主要类型：

| 算法 | 核心思想 | 复杂度 | 灵活性 | 典型应用 |
|------|---------|--------|--------|---------|
| **有限状态机（FSM）** | 状态 + 转移条件 | ⭐ | ⭐ | 简单NPC巡逻 |
| **层次状态机（HFSM）** | 状态嵌套、共享子状态 | ⭐⭐ | ⭐⭐ | 角色控制 |
| **行为树（BT）** | 树状节点编排行为 | ⭐⭐ | ⭐⭐⭐ | 3A游戏AI标准 |
| **Utility 系统** | 实时评分选择 | ⭐⭐⭐ | ⭐⭐⭐⭐ | 复杂决策 |
| **HTN 规划器** | 任务分解为子任务 | ⭐⭐⭐ | ⭐⭐⭐ | 战术规划 |
| **GOAP** | 目标驱动的行动规划 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 战术AI |

**选型趋势：**
- 2010年前：FSM 为主流
- 2010-2020：行为树成为行业标准
- 2020年后：行为树 + Utility 混合成为主流方案

### 结构架构模式

"结构架构"关心的是 AI 代码如何组织在项目中。

| 模式 | 说明 | 适合 |
|------|------|------|
| **分层架构** | AI 分为感知层→决策层→行动层 | 中大型项目 |
| **组件架构** | AI 能力拆分为独立组件（感知组件、决策组件） | 任何规模 |
| **消息驱动** | 组件间通过消息通信 | 解耦需求高的项目 |
| **数据驱动** | AI 行为由数据（配置表/脚本）定义 | 需要频繁调整的游戏 |
| **服务架构** | 寻路服务、感知服务、决策服务各自独立 | 大型团队 |

## 架构选型决策矩阵

```
你的项目是？
│
├── 小型手游/原型
│   └── FSM 或 轻量行为树（足够用，别过度设计）
│
├── 中大型单机/网游
│   ├── NPC 逻辑复杂？→ 行为树 + 黑板
│   ├── 需要战术决策？→ 行为树 + Utility 融合
│   └── 开放世界？→ 分层架构 + LOD AI 系统
│
├── RTS / 策略游戏
│   ├── 大量单位？→ 服务架构 + 行为树简化版
│   └── 高级战术？→ HTN 或 GOAP
│
└── 叙事/角色扮演
    └── 行为树 + 社交模拟系统
```

### 架构分层示例（中大型项目）

```
┌─────────────────────────────────┐
│        感知层 (Perception)       │ ← 视觉/听觉/记忆
│   视锥检测 · 声音事件 · 目标追踪  │
├─────────────────────────────────┤
│        决策层 (Decision)          │ ← 行为树/Utility
│   行为选择 · 目标设定 · 策略评估  │
├─────────────────────────────────┤
│        行动层 (Action)            │ ← 移动/攻击/交互
│   寻路 · 动画 · 武器系统 · 对话  │
├─────────────────────────────────┤
│        基础设施 (Infrastructure)  │ ← 服务/工具
│   寻路服务 · 感知服务 · 黑板共享  │
└─────────────────────────────────┘
```

## 实战要点

### 1. AI 与游戏逻辑的解耦

AI 系统和游戏逻辑之间的接口应该是**干净的 API**，而不是直接调用：

```python
# ❌ 坏实践：AI 直接操控游戏对象
ai.grenade.launch(target)
health -= damage

# ✅ 好实践：AI 发出指令，游戏系统执行
ai.emit("request_action", ActionType.GRENADE, target)
# → CombatSystem 接收并执行
```

### 2. AI LOD（细节层次）

不同距离的 AI 使用不同的决策精度：

| 距离 | Tick率 | 行为树深度 | 感知范围 | 寻路精度 |
|------|--------|-----------|---------|---------|
| 近（0-15m） | 15-30Hz | 完整 | 全 | 高精度 |
| 中（15-50m） | 5-10Hz | 简化 | 部分 | 粗略 |
| 远（>50m） | 1-3Hz | 最简 | 无 | 直接移动 |

### 3. 数据驱动 Design

将 AI 参数放在配置文件而非代码中：

```python
# ai_config.json
{
    "patrol": {
        "speed": 2.5,
        "alert_range": 15.0,
        "tick_rate": 10
    },
    "combat": {
        "preferred_weapon": "rifle",
        "cover_preference": 0.7,
        "aggression": 0.6
    }
}
```

## 反模式（NEVER 列表）

| # | 反模式 | 原因 |
|---|--------|------|
| 1 | ❌ AI 与渲染/表现逻辑耦合 | AI 决策不应依赖动画时间线或特效状态 |
| 2 | ❌ 所有 AI 用同一种架构 | 小兵和 Boss 的复杂度不同，架构应匹配 |
| 3 | ❌ 过度设计 | 原型阶段用 FSM 够了，不要一开始就上 HTN |
| 4 | ❌ AI 逻辑分散在各处 | 一个 AI 的决策代码应集中在一个可管理的范围内 |
| 5 | ❌ 忽略数据驱动 | 硬编码的 AI 参数需要重新编译才能修改 |
