# 案例1：以迭代设计流程开发一个新游戏

> 使用案例。**以迭代设计为主线**——不是按功能（数值/AI/寻路）平铺，而是跟随 Macklin & Sharp 的四步循环「概念化 → 原型 → 玩测 → 评估」，在循环的每一圈融入 wiki 的对应 concept。假设场景：开发一款**单按钮动作小游戏**。

> [!info] 为什么以迭代设计为主线
> 游戏是**二阶设计**（设计师只定义参数，体验要等玩家来玩才成型），无法"想清楚再做对"。[[迭代设计流程]] 的核心信条：**体验无法靠想象判断，必须靠玩**。所以开发不是线性瀑布，而是反复"做-测-改"的螺旋上升。

---

## 循环骨架（贯穿全程）

```
        ┌──────────────────────────────────┐
        │                                  ▼
    概念化 ──→ 原型 ──→ 玩测 ──→ 评估 ──→ 回到概念化
```

每一圈循环：
1. **概念化**：这一圈要回答什么问题？锁定设计价值。
2. **原型**：做最粗糙的可玩原型来回答那个问题。
3. **玩测**：找人玩，观察真实体验。
4. **评估**：体验达标吗？下一个最高风险的问题是什么？

> [!tip] 迭代铁律（来自 [[迭代设计流程]] + [[原型设计]]）
> - **每个原型只回答一个问题**
> - **每次循环只改一个变量**（否则无法归因结果）
> - **尽早失败**（early failure is nutrient）——粗糙原型越早，修正成本越低

---

## 第 1 圈：确立体验目标与核心乐趣

### 概念化

- 读 [[游戏体验设计]]：先剥离游戏结构，问"我想让玩家有什么核心感受"。游戏≠体验，设计师创造的是体验。
- 读 [[透镜方法]]：用透镜追问式审视（如"本质体验透镜""兴趣曲线透镜"）。
- 读 [[兴趣曲线]]：预想一次游玩的时间-兴趣曲线，要有钩子→铺设→高潮。
- 读 [[小游戏哲学]]（joys）：小游戏（10分钟内、浏览器可玩）有独立魅力，不是大游戏踏板。

**这一圈要回答**：玩家 90 秒内的核心感受是什么？（如"心流与紧张释放的循环"）

### 原型

- 读 [[约束激发创意]]：施加约束催生独特机制。单按钮（interaction 约束）迫使设计独特动作。
- 读 [[单按钮游戏设计]]：单按钮机制四大分类——瞬移/分裂/选择/属性反转。
- 读 [[原型设计]]：**先构建玩具**——先确认"它好玩吗"再考虑系统完整性。做纸原型或物理原型。

**这一圈原型**：一个最简可玩 demo（< 200 行），只验证"核心动作好不好玩"。

### 玩测

- 读 [[迭代设计流程]]：区分"目标受众玩测"与"朋友家人玩测"。
- 观察：玩家是否自发地"再来一次"？核心动作是否产生预期的紧张/释放？

### 评估

- 用 [[游戏体验设计]] 的"体验目标"校准：观察到的是否接近预期？
- **决策**：核心机制成立 → 进入第 2 圈；不成立 → 回到概念化换机制。

---

## 第 2 圈：验证核心循环的风险点

### 概念化

- 读 [[迭代设计流程]]：**风险评估**——列出所有不确定点，最高风险先原型化。
- 对单按钮游戏，最高风险通常是"单按钮能否撑起足够变化"。

### 原型

- 读 [[原型设计]]：八种原型——这里可能需要"代码/技术原型"验证可行性，或"可玩原型"验证节奏。
- 读 [[程序化生成]]（如需要变化源）：随机点变异/标签组合生成内容变体。

### 玩测

- 读 [[迭代设计流程]]：用"目标受众"玩测，记录具体行为（停留时长、失败点、情绪）。

### 评估

- **决策**：哪些变量需要进入下一圈的数值调校？（如速度、生成频率、惩罚力度）

---

## 第 3 圈：数值平衡与难度曲线

### 概念化

- **这一圈要回答**：难度是否落在心流区间？有没有碾压策略？

### 原型（数值也是原型）

- 读 [[难度曲线设计]]：连续曲线 `sqrt(frame*0.0001)+1` + 锯齿波，或离散关卡 `pow(random(),100/(lvl+1))`。关键是难度提升不能单调。
- 读 [[数值平衡]]：分层平衡模型——先定"生存周期常数"（预期单场回合数），反推攻防血比。
- 读 [[数值公式体系]] / [[成长曲线设计]]：伤害公式、属性转化、养成曲线的渐近行为。
- 用 Machinations 或 VBA 模拟跑千次战斗验证（[[数值平衡]] 的"模拟验证"原则）。

### 玩测

- 读 [[难度曲线设计]]：判断标准——玩家失误时**归因于自己技术** → 合适；感到**不公** → 不当。
- 优先用"游戏速度"调难度（玩家与敌人同比例提速，最不易不公）。

### 评估

- **决策**：曲线参数定稿；进入 AI/敌人行为圈（如有 NPC）。

---

## 第 4 圈：AI 决策（如游戏有 NPC/敌人）

### 概念化

- **这一圈要回答**：敌人行为是否灵活可信？有没有碾压/被碾压的 AI 策略？

### 原型

- 读 [[游戏AI架构]]：选型——FSM / 行为树 / Utility / HTN。
- 读 [[游戏AI行为树]]：Sequence/Selector/黑板，当前工业最广泛架构。
- 读 [[效用理论]]：当行为树选择逻辑变复杂时，Utility 评分 + 噪声注入让行为自然多变。
- 决策矩阵：

| 敌人需求 | 推荐方案 |
|---|---|
| 简单巡逻-警戒-攻击 | 行为树 |
| 灵活选择武器/战术 | 行为树 + Utility |
| 复杂多步骤战术 | HTN（[[HTN层次任务规划]]） |
| 棋类/回合制对抗 | [[博弈树搜索]]（Minimax/MCTS） |

### 玩测

- 读 [[智能的错觉]]：**AI 看起来聪明比真的聪明更重要**。玩测看玩家感知，不看 AI 内部复杂度。
- 读 [[战术推理]]：若需掩体/包抄，用战术位置评分。

### 评估

- **决策**：AI 行为是否产生有趣的对抗节奏？进入打磨圈。

---

## 第 5 圈：移动与寻路（如需空间移动）

### 概念化 / 原型

- 读 wiki 待补充的寻路系统：NavMesh + A* 优化。
- 读 [[向量与运动]] + [[几何与碰撞]]：移动与碰撞的数学基础。
- 读 [[群体行为]]：多单位用 Boids/转向力。

### 玩测 / 评估

- 单位是否卡墙？路径是否自然？碰撞是否公平？

---

## 第 6 圈：游戏手感打磨（Juiciness）

### 概念化

- **这一圈要回答**：操作的每个瞬间是否有足够的反馈冲击？

### 原型

- 读 [[Juiciness与游戏手感]]：
  - **冲击感**：震屏 + hit-stop（受击停顿）+ 粒子 + 击退。
  - **生命感**：物体加眼睛/表情，对环境反应（低成本高回报）。
  - **forgiveness**：coyote time（离台仍可跳）、输入缓冲（提前按键生效）——宽恕玩家小失误。
- 读 [[游戏体验设计]]：juice 是塑造体验的视听手段。

### 玩测

- 对比"加 juice 前/后"，同样的规则是否变得有趣得多？

### 评估

- **决策**：手感达标，进入发布前的最终验证。

---

## 第 N 圈：收敛与发布

### 概念化

- 读 [[迭代设计流程]]：用"设计价值"过滤"好但不适合本作"的想法，防止范围蔓延。
- 读 [[约束激发创意]]：标准化流程加速完成（固定工具/类型/流程，精力集中在核心创新）。

### 玩测（最终）

- 读 [[迭代设计流程]]：六步评估法（复查-孵化-头脑风暴-决定-编辑文档-日程）。
- 读 [[智能的错觉]]：最终判据是玩家感知，不是内部指标。

### 评估 → 发布

- 迭代没有真正"完成"，只有"足够好就发布"。发布后继续迭代（见 [[实践方法论]] 的 Game A Week 理念）。

## 迭代循环中的 wiki 知识全景

```
概念化  ← 游戏体验设计 / 透镜方法 / 兴趣曲线 / 小游戏哲学 / 风险评估
   │
   ▼
原型    ← 约束激发创意 / 单按钮设计 / 原型设计 / 程序化生成
   │      难度曲线 / 数值平衡 / 数值公式 / 行为树 / 效用理论 / 寻路
   │      Juiciness / 向量运动 / 几何碰撞 / 群体行为
   │
   ▼
玩测    ← 迭代设计流程(玩测分类) / 难度曲线(归因标准) / 智能的错觉(玩家感知)
   │
   ▼
评估    ← 游戏体验设计(体验目标校准) / 迭代设计流程(六步评估法) / 设计价值过滤
   │
   └──→ 回到概念化（下一个最高风险问题）
```

> [!warning] 迭代的三大反模式（来自 [[迭代设计流程]] / [[原型设计]]）
> 1. ❌ **想清楚再做对**（预测式）——游戏是二阶设计，体验必须靠玩验证
> 2. ❌ **润色是迭代的敌人**——过早打磨会让人舍不得推翻有问题的核心
> 3. ❌ **一次改多个变量**——无法归因玩测结果
