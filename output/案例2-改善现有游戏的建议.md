# 案例2：从 wiki 获取改善现有游戏的建议

> 使用案例。假设你**已经有一个游戏**，但体验有问题。本案例按常见痛点给出"症状 → 诊断 → wiki 出处 → 改进方案"的对照表，对症下药。

---

## 痛点 A：敌人/NPC 感觉很蠢、行为机械

**症状**：NPC 行为可预测、不会变通；玩家觉得"AI 像机器不像活物"。

**诊断**：AI 决策架构过于刚性（纯 FSM 或硬编码），缺乏灵活性与"智能错觉"。

**wiki 出处与方案**：

| 诊断 | 出处 | 改进 |
|---|---|---|
| 行为太确定、可预测 | [[效用理论]] | 引入 Utility 评分 + 噪声注入，让行为自然多变（非每次都选最优） |
| 决策架构难扩展 | [[游戏AI架构]] / [[游戏AI行为树]] | 改用行为树（模块化、可组合）或行为树+Utility 混合 |
| AI 全知、反应过快 | wiki 待补充的感知系统 | 加感知过滤 + 反应时间延迟，AI 只知道"该知道的" |
| AI 看起来不像在思考 | [[智能的错觉]] | 八大技巧营造智能错觉——玩家觉得聪明比真的聪明更重要 |
| 战斗行为单调 | [[战术推理]] | 加战术位置评分（掩体/包抄），让敌人选"更好的位置"而非冲脸 |

**优先级**：先做"智能错觉"（低成本高回报），再做架构升级。

---

## 痛点 B：数值失衡、有碾压策略

**症状**：存在"最优解"碾压一切；某些属性/武器明显过强或过弱；后期数值爆炸。

**诊断**：数值平衡缺失，未消除"策略退化"（dominant strategy）。

**wiki 出处与方案**：

| 诊断 | 出处 | 改进 |
|---|---|---|
| 属性价值不可比 | [[数值平衡]] | 用"属性价值汇率法"（攻击看 DPS 增量，防御看 EHP 增量）统一度量 |
| 战力计算不准 | [[战斗数值设计]] / [[数值公式体系]] | 建立标准人基准 + 战力计算公式，分层校验 |
| 平衡靠拍脑袋 | [[数值平衡]] | 用 Machinations 可视化建模 + VBA/模拟器跑千次战斗验证 |
| 后期数值爆炸 | [[成长曲线设计]] | 检查成长函数的渐近行为，用对数/sqrt 抑制后期增长 |
| 经济系统崩溃 | [[虚拟经济系统]] / [[货币与市场]] | 用市场结构 + 水龙头/污水槽调控，避免禁令催生黑市 |

**优先级**：先定"生存周期常数"（预期单场战斗回合数），反推攻防血比。

---

## 痛点 C：难度曲线陡峭、玩家挫败流失

**症状**：新手劝退、老玩家无聊；难度要么跳崖要么平淡。

**诊断**：难度曲线设计不当，未匹配心流区间。

**wiki 出处与方案**：

| 诊断 | 出处 | 改进 |
|---|---|---|
| 难度单调线性上升 | [[难度曲线设计]] | 用 sqrt 曲线（后期平缓）+ 锯齿波（周期性骤降制造波动） |
| 玩家感到"不公" | [[难度曲线设计]] | 检查是否有"不可规避攻击"；玩家应归因于技术而非不公 |
| 早期太易没紧张感 | [[难度曲线设计]] | 引入风险回报机制（贴近危险得奖励分），从早期就制造紧张 |
| 关卡难度无变化 | [[难度曲线设计]] | 离散关卡用 `pow(random(),100/(lvl+1))`，制造有波动的递进 |
| 无法判断难度是否合适 | [[难度曲线设计]] | 工具 diffi-tween 可视化调曲线 + 即时试玩 |

**优先级**：优先用"游戏速度"调难度（玩家与敌人同比例提速，最不易不公）。

---

## 痛点 D：游戏手感干涩、缺乏反馈

**症状**：操作不顺滑、击中没有冲击感、画面死板。

**诊断**：缺乏 Juiciness（多汁）与 forgiveness（宽恕）。

**wiki 出处与方案**：

| 诊断 | 出处 | 改进 |
|---|---|---|
| 击中/得分没有冲击感 | [[Juiciness与游戏手感]] | 加震屏 + hit-stop（受击停顿）+ 粒子 + 击退动画 |
| 移动/动画生硬 | [[Juiciness与游戏手感]] | 用 tweening/easing 平滑动画，物体大小随事件弹跳变化 |
| 物体没有"生命感" | [[Juiciness与游戏手感]] | 给物体加眼睛/微笑/表情，让它对环境反应（低成本高回报） |
| 操作容错低、常误操作 | [[Juiciness与游戏手感]] | 加 forgiveness：coyote time（离台仍可跳）、输入缓冲（提前按键生效） |
| 缺乏听觉反馈 | [[Juiciness与游戏手感]] | 响应式音效（每个事件配声），可参考 [[程序化生成]] 用 sfxr 生成音效 |

**优先级**：表情化物体 + 粒子 + 响应式音效是最低成本的 juice 三件套。

---

## 痛点 E：留存差、玩家玩几次就走

**症状**：次日留存低、没有持续游玩动力。

**诊断**：缺乏玩家动机设计 + 叙事节奏 + 长期目标。

**wiki 出处与方案**：

| 诊断 | 出处 | 改进 |
|---|---|---|
| 不知道玩家为何流失 | [[玩家动机与留存]] | 识别玩家动机类型（成就/社交/沉浸），对准设计 |
| 玩家没有长期目标 | [[情感工程]] / [[成长曲线设计]] | 设计强化程序（变比强化最有效但有成瘾争议）+ 可见成长 |
| 体验节奏平 | [[游戏体验设计]] | 用兴趣曲线检查节奏，确保有钩子→铺设→高潮 |
| 缺乏新鲜感 | [[程序化生成]] | 程序化关卡/素材生成，提升重玩价值 |
| 难度劝退 | （见痛点 C） | 调难度曲线到心流区间 |

**优先级**：留存是系统性问题，需组合改进，单点优化效果有限。

---

## 通用改进方法论

无论哪个痛点，遵循这个循环：

```
1. 玩测观察（记录玩家实际行为，不问"好不好玩"）
       ↓
2. 定位症状（对照上方痛点表，找到最匹配的诊断）
       ↓
3. 查 wiki 出处（读对应 concept 的方法/模型）
       ↓
4. 小步改进（每次只改一个变量，见 [[迭代设计流程]]）
       ↓
5. 再玩测验证（确认改进有效，无新问题）
       ↓
6. 回到 1
```

> [!warning] 常见陷阱
> - 不要一次改太多（无法归因）
> - 不要只信纸面推导（数值/AI 必须实测模拟）
> - 不要追求"完美平衡"（平衡是消除碾压策略，不是让一切等强）
