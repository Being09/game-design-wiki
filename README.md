# 游戏设计知识库

> 把 **17 本游戏设计与 AI 教材** + **一线独立开发者的实战方法论**，编译成一个可导航、可查询的中文知识网络。

无论你是想**从零设计一个游戏**，还是**改进手头的游戏**，还是**查一个具体的设计问题**——这里有经过提炼的答案，而非散落各处的原始资料。同一个主题往往同时给出教材的抽象模型与实践的具体手法，让你既懂"为什么"又懂"怎么做"。

本库由 [llm-wiki](https://github.com/nvk/llm-wiki)（LLM 编译知识库框架）驱动构建。

## 你能用它做什么

### 🎮 场景一：我要做一个新游戏

跟着 [案例1：新游戏设计开发流程](output/案例1-新游戏设计开发流程.md) 走——它以**迭代设计**的四步循环（概念化→原型→玩测→评估）为主线，6 圈迭代螺旋推进：

```
确立体验目标 → 验证核心乐趣 → 数值难度 → AI 决策 → 寻路移动 → 手感打磨
```

每一圈都告诉你"读哪个 concept、做什么原型、玩测看什么、如何决策"。从"我想做一个单按钮动作小游戏"到可发布的成品，路径完整。

### 🔧 场景二：我要改进现有游戏

看 [案例2：改善现有游戏的建议](output/案例2-改善现有游戏的建议.md)——按 5 大常见痛点对症下药：

| 你的游戏的问题 | 本库能给你 |
|---|---|
| 敌人/NPC 感觉很蠢 | 让 AI 行为自然多变的架构选型 + 营造"智能错觉"的技巧 |
| 数值失衡、有碾压策略 | 分层平衡模型 + 模拟验证方法 |
| 难度太陡或太无聊 | 难度曲线公式（sqrt/锯齿波）+ 心流区间判断 |
| 手感干涩、缺乏冲击 | Juiciness 手法 + forgiveness 宽恕机制 |
| 玩家留存差 | 玩家动机设计 + 兴趣曲线 + 强化程序 |

### 🔍 场景三：我只想查一个问题

直接去 `wiki/topics/` 找对应领域页，或全局搜索关键词。比如：
- "行为树怎么设计" → `wiki/concepts/游戏AI行为树.md`
- "难度曲线怎么调" → `wiki/concepts/难度曲线设计.md`
- "怎么让游戏更好玩" → `wiki/concepts/Juiciness与游戏手感.md`

> [!tip] 把案例当知识地图
> 案例里的每个 `[[概念名]]` 都是可点击链接。遇到不熟的概念，顺着链接进 concept 看完整定义，再回到案例——这是最高效的学习路径。

## 怎么用

本库是纯 Markdown + YAML frontmatter，**无任何专有格式或运行时依赖**。两类用户各取所需：

### 👤 人类阅读（推荐 Obsidian 或 VS Code）

| 工具 | 为什么选它 |
|---|---|
| **[Obsidian](https://obsidian.md/)**（首选） | 图谱视图可视化概念间的关联、反向链接面板看"谁引用了这个概念"、本地优先无云端依赖 |
| **[VS Code](https://code.visualstudio.com/)** | 内置 Markdown 预览 + 文件树，适合熟悉代码的用户 |

**3 步上手：**
1. 用 Obsidian 打开本目录作为 Vault（或 VS Code 打开文件夹）
2. 从 `wiki/_index.md` 开始浏览，点 `[[双链]]` 在概念间跳转
3. 想查某主题 → 找 `wiki/topics/<领域>.md` 领域页，或全局搜索（`Ctrl+Shift+F`）

### 🤖 LLM Agent（不受限）

任何能读写 Markdown 的 Agent（Claude Code、Codex、Gemini、Cursor）都可独立查询，README 的[附录 A](#附录-a-llm-查询操作手册) 是详细操作手册。要点：

- **先读索引，再读正文**：`_index.md` 是导航，`.md` 文件是真相
- **从领域页切入**：`wiki/topics/<领域>.md` 汇总该领域的所有 concepts
- **跟随 See Also**：每个 concept 的 See Also 链接理论↔实践互补视角，务必顺读

## 附录 A：LLM 查询操作手册

> 核心原则：**索引是导航，文件是真相**。先读索引定位，再读正文取证。

**标准查询流程：**
```
1. wiki/_index.md       → 了解全库（三个子目录计数）
2. topics/<领域>.md      → 进入对应领域导航页
3. concepts/<概念>.md    → 读最相关的综合文章
4. concept 的 See Also   → 顺链接找关联主题
5. references/<书名>.md  → 需要单书提炼时（含外部原著链接）
6. references 的外部链接 → 需查阅原著全文时
7. Grep                  → 全文搜索补充
```

**frontmatter 字段：** 每个 `.md` 有结构化元数据——`title`/`category`/`tags`/`related`（关联概念）/`sources`（来源）/`confidence`（high|medium|low）/`synthesized`（日期）。回答时标注 confidence，遇 `low` 提示需复核。

**双链格式：** `[[slug|中文名]] ([中文名](../concepts/slug.md))`——`[[ ]]` 供 Obsidian 图谱，`(path)` 供标准 Markdown 工具导航。

## 附录 B：目录结构

```
game-design/
├── README.md              # 本文件
├── config.md              # 库元信息（来源、领域划分）
├── log.md                 # 操作日志（只追加）
├── wiki/                  # 编译后的知识层
│   ├── references/        # 单源提炼（每本书/每大章一篇，标注外部来源）
│   ├── concepts/          # 多源综合（跨 2+ 来源的主题文章）
│   └── topics/            # 领域导航页
└── output/                # 应用产物（使用案例等）
```

**分层语义：** `references/` 单源提炼 → `concepts/` 多源综合 → `topics/` 领域导航。`_index.md` 是派生缓存，手动维护（按 frontmatter 重建）。原文档案已删除（见 config.md 合规化说明），各文章标注外部来源链接供溯源。
